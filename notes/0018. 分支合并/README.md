## 1. 🎯 本节内容

- 快进合并（Fast-forward）
- 三方合并与合并提交（Merge Commit）
- 合并命令（`git merge`）

## 2. 🫧 评价

合并 ≠ 复制改动的内容到同一个文件中（这是我们通过本地多文件的形式来做版本管理的时候采用的原始方案，Git 的分支合并并不是这么做的）

合并 = 寻找共同祖先 + 计算两路差异 + 智能应用差异 + 解决逻辑冲突 + 生成带双父指针的新快照

合并分支的核心目的，就是将不同开发线（分支）上的独立改动，安全、有序地集成到一个统一的节点（通常是主分支的最新提交）上，从而形成一条完整、连续的开发历史。

## 3. 🤔 如何使用 git merge 命令？

```bash
# 将 feature 分支合并到当前分支
git switch main
git merge feature

# 合并时禁止快进（始终创建合并提交）
git merge --no-ff feature

# 合并时附带提交消息
git merge feature -m "合并 feature 分支到 main"

# 只检查能否合并，不实际执行
git merge --no-commit feature
git merge --abort  # 取消合并

# 使用 squash 合并（将分支的多次提交压缩为一次）
git merge --squash feature
git commit -m "合并 feature 功能"
```

`--squash` 合并不会创建合并提交，而是将目标分支的所有更改合并到暂存区，需要手动提交。这种方式可以保持主分支历史的整洁。

合并操作的最佳实践：

- 合并前确保工作区干净（无未提交的修改）
- 合并前先拉取远程最新代码
- 重要分支的合并建议使用 `--no-ff` 以保留分支历史

## 4. 🤔 什么是快进合并？

快进合并（Fast-forward）是最简单的合并方式。当目标分支是当前分支的直接上游（即当前分支的提交历史是目标分支的子集）时，Git 只需将当前分支指针向前移动即可，不需要创建额外的合并提交。

```bash
# 假设 feature 分支比 main 分支领先若干次提交
git switch main
git merge feature
# 输出: Fast-forward
```

合并前：

```
C2（main）
  \
   C3 -> C4（feature）
```

快进合并后：

```
C2
  \
   C3 -> C4（main，feature）
```

如果想要在快进合并时也生成一个合并提交（保留分支历史），可以使用 `--no-ff` 参数：

```bash
git merge --no-ff feature
```

合并后：

```text
      C3 --- C4 (feature)
     /       /
... C2 ----- M (main, feature merged)
             ^
             |
         (合并提交 M)

执行 git merge --no-ff feature 后：
Git 强制创建一个新的“合并提交” (M)，即使可以快进。
main 指针移动到 M，feature 依然指向 C4。
分支的历史分叉线被永久保留在日志中。
```

## 5. 🤔 什么是三方合并与合并提交？

当两个分支各自有了新的提交，形成了分叉，Git 就无法通过快进合并来完成。此时 Git 会执行三方合并（three-way merge），使用两个分支的最新提交以及它们的共同祖先来创建一个新的合并提交（Merge Commit）。

```
          C3 -> C5 (main)
         /
C1 -> C2
         \
          C4 -> C6 (feature)
```

合并后：

```
          C3 -> C5 ---\
         /             \
C1 -> C2                C7 (main, merge commit)
         \             /
          C4 -> C6 ---/
```

合并提交有两个父提交，记录了两条分支的汇合点。

合并过程中如果有出现冲突，需要手动解决冲突，然后再次提交才能完成合并。

## 6. 🤔 分支合并时出现冲突如何处理？

当 Git 无法自动合并两个分支对同一文件的同一行（或相邻行）做出的不同修改时，就会触发冲突。此时 `git merge` 会暂停，等待用户介入。

处理步骤：

识别冲突：Git 会在终端提示 `CONFLICT (content): Merge conflict in <filename>`，并将冲突标记直接写入文件内容中。

```text
<<<<<<< HEAD
这是当前分支 (main) 的代码内容
=======
这是要合并分支 (feature) 的代码内容
>>>>>>> feature
```

手动编辑：打开冲突文件，根据业务逻辑决定保留哪部分代码，或者将两者结合。

- 保留 `HEAD`：删除分隔线和 `feature` 部分。
- 保留 `feature`：删除分隔线和 `HEAD` 部分。
- 混合保留：手动编写新代码，并务必删除 `<<<<<<<`, `=======`, `>>>>>>>` 这些标记符号。

在一些编辑器（比如 VS Code）中，会自动识别上述这样的标记符号，并提供相关按钮来快速处理这些冲突的内容。

![img](https://cdn.jsdelivr.net/gh/tnotesjs/imgs-2026@main/2026-03-13-13-30-12.png)

标记解决：修改完成后，使用 `git add` 将文件标记为“已解决”。

```bash
git add <conflicted_file>
# 或者一次性添加所有已解决的文件
git add .
```

完成提交：执行 `git commit`。

- 如果是正常的 `git merge` 流程，Git 会自动生成默认的合并提交信息（如 `Merge branch 'feature'`），直接保存即可。
- 此时，新的合并提交（如上例中的 `C7`）正式生成，拥有两个父节点（`C5` 和 `C6`），分叉的历史正式汇合。

```bash
git commit
```

合并成功后的最终状态：

```text
          C3 -> C5 (main 原头)
         /       \
C1 -> C2          C7 (新的合并提交，有两个父节点: C5 和 C6)
         \       /
          C4 -> C6 (feature 原头)
```

指针状态:

- main 指向 C7
- feature 依然指向 C6 (除非你随后删除了该分支)

## 7. 🤔 合并的本质是什么？

合并（Merge）的本质绝对不是“复制粘贴”代码内容，而是“基于共同祖先，计算差异并应用差异”的数学/逻辑过程。

如果仅仅是“复制内容”，Git 就无法处理冲突，也无法生成正确的历史拓扑。

下面让我们深入拆解一下合并的真实本质：

### 7.1. 核心机制：三方合并算法（Three-Way Merge）

Git 在执行合并时，内部实际上是在做一场“找不同”的游戏。它不直接比较两个分支的文件，而是引入第三个关键角色：共同祖先（Common Ancestor / Base）。

假设我们要合并 `main C5` 和 `feature C6`，它们的共同祖先是 `C2`。

Git 的逻辑是这样的：

- 对比 A：计算 `C2` 祖先 $\to$ `C5` main 发生了哪些变化？（记为 $\Delta_A$）
  - 例如：第 10 行由 "Hello" 变成了 "Hi"。
- 对比 B：计算 `C2` 祖先 $\to$ `C6` feature 发生了哪些变化？（记为 $\Delta_B$）
  - 例如：第 20 行由 "World" 变成了 "Git"。
- 智能应用：
  - 如果 $\Delta_A$ 和 $\Delta_B$ 修改的是不同的行或文件 $\to$ 自动合并（同时应用两个变化）。
  - 如果 $\Delta_A$ 和 $\Delta_B$ 修改的是同一行的不同内容 $\to$ 标记冲突（因为 Git 无法判断该听谁的）。
  - 如果 $\Delta_A$ 和 $\Delta_B$ 修改的是同一行的相同内容 $\to$ 视为无冲突（殊途同归）。

结论：合并是差异（Diff）的运算与重组，而不是文件的简单覆盖或复制。

### 7.2. 合并提交的本质：一个特殊的“快照”

当你执行完合并，生成的那个合并提交（Merge Commit），它的本质是什么？

- 合并提交包含一个完整的文件树快照（Tree Object）：这个快照是上述“差异运算”后的最终结果。
- 合并提交拥有两个父节点指针：这是它与普通提交最大的不同。普通提交只有一个父节点（线性历史），而合并提交有两个（分叉历史的汇合点）。
  - 父节点 1：通常是当前分支的上一个提交。
  - 父节点 2：是被合并分支的最后一个提交。
- 合并提交是一个基于上下文的智能集成过程，旨在保留两个分支独立开发的成果，并将它们逻辑自洽地统一起来，而绝非简单的文件内容复制。
