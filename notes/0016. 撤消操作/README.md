## 1. 🎯 本节内容

- 补充提交（`git commit --amend`）
- 取消暂存的文件（`git restore --staged` / `git reset HEAD`）
- 撤消对文件的修改（`git restore` / `git checkout --`）
- 使用 `git revert` 进行安全撤销

## 2. 🫧 评价

回顾日常使用 Git 的场景，会发现使用到撤消操作的场景并不是很多。

个人用到用的最多的是 `git commit --amend -m "修正后的提交消息"` 重写 commit message。

实际在使用过程中，如果发现有啥文件内容需要回退的话，一般会去查阅 Git 的历史记录，找到自己需要的内容版本，然后再手动搬运需要回退的内容。倘若改动的量很大，才有可能会考虑使用本节提到的 `git restore` 命令来回退整个版本。

## 3. 🤔 如何补充/重写提交信息？

如果提交后发现漏掉了文件或提交消息写错了，可以使用 `--amend` 参数来修补上一次提交：

```bash
# 修改提交消息
git commit --amend -m "修正后的提交消息"

# 补充遗漏的文件到上一次提交
git add forgotten_file.txt
git commit --amend --no-edit
```

- `--amend` 实际上会用一个新的提交替换掉上一个提交，这意味着再次执行 commit 之后：
  - 上一次提交的内容会被新的提交覆盖（如果暂存区有内容，这些内容会被包含在新提交中）
  - 上一次提交的 SHA-1 值会改变，新的提交会有一个新的 SHA-1 值
- `--no-edit` 参数表示不修改提交消息。

注意：不要对已经推送到远程仓库的提交使用 `--amend`，因为这会改变提交的 SHA-1 值，导致历史不一致。

## 4. 🤔 如何取消暂存的文件？

如果不小心将不需要的文件添加到了暂存区，可以取消暂存：

::: code-group

```bash [推荐方式]
# 推荐方式 --staged （Git 2.23+）

# 取消特定文件的暂存
git restore --staged file.txt
# --staged 参数的作用是告诉 Git：“我要操作的对象是暂存区（Index），而不是工作目录”
# 如果不加这个参数，git restore file.txt 会丢弃你在工作目录中的修改（危险操作）
# 加上 --staged，它只把文件从暂存区移除，保留工作目录的修改

# 取消所有暂存
git restore --staged .
# . 参数代表当前目录下的所有文件
# 这条命令会将当前目录下所有已暂存的文件全部移出暂存区
```

```bash [传统方式]
# 传统方式

# 取消特定文件的暂存
git reset HEAD file.txt
# 在 Git 2.23 之前，这是唯一的标准做法
# 虽然现在有了 restore，但 reset 依然被广泛使用，且兼容所有 Git 版本
# HEAD 是一个引用，指向当前分支的最新一次提交
# 这里的逻辑是：“将暂存区中 file.txt 的状态，重置为和 HEAD（即上一次提交时）一样的状态”
# 因为文件在上一次提交后可能还没被暂存过，或者我们就是想把它从“准备提交”列表中拿掉，所以这等同于“取消暂存”
# 注：在现代 Git 中，HEAD 甚至可以省略，直接写 git reset file.txt 效果一样，但写上 HEAD 语义更清晰

# 取消所有暂存
git reset HEAD .
# 或者简写为
git reset
# 如果不指定文件名，git reset 默认会对所有已暂存的文件进行操作（相当于 .）
```

:::

上述提到的这个取消暂存的操作只是将文件从暂存区移除，不会影响工作区中的文件内容。

实际操作流程示例 -> 假设你修改了 `app.js` 并添加到了暂存区，现在想把它拿出来单独处理：

```bash
# 1. 查看状态，看到 app.js 在 "Changes to be committed"
git status

# 2. 执行取消暂存 (任选其一)
git restore --staged app.js
# 或者
# git reset HEAD app.js

# 3. 再次查看状态
git status

# 此时，你会发现 app.js 出现在 "Changes not staged for commit" 列表中。
# 这意味着它依然是修改过的，但下次 git commit 时不会包含它，除非你再次 git add。
```

| 特性 | `git restore --staged` | `git reset HEAD` |
| :-- | :-- | :-- |
| 引入版本 | Git 2.23 (2019年) | 所有 Git 版本 |
| 设计目的 | 专门用于恢复文件状态（暂存区或工作区），职责单一。 | 主要用于重置分支指针，兼管暂存区，功能较多。 |
| 易读性 | 高。`--staged` 明确指出了操作目标是暂存区。 | 中。需要理解 `reset` 和 `HEAD` 的关系。 |
| 安全性 | 较高。参数意图明确，不易误操作覆盖文件。 | 需谨慎。`git reset --hard` 会丢失修改，新手容易混淆。 |
| 推荐场景 | 新项目、团队统一使用新版 Git、追求语义清晰。 | 维护旧脚本、需要在老旧服务器上操作、个人习惯。 |

⚠️ 常见误区提醒 -> 很多初学者会混淆以下两个命令，请务必注意区别：

- 取消暂存（你想要的）：
  - `git restore --staged file.txt`
  - 后果：文件还在，修改还在，只是不提交了。
- 丢弃修改（危险！）：
  - `git restore file.txt` (没有 `--staged`)
  - 后果：文件会被还原到最后一次提交的状态，你刚才写的代码全丢了！
  - 如果是想丢弃修改，通常建议先用 `git stash` 备份，确认不要了再还原。

## 5. 🤔 如何撤消对文件的修改？

如果想要丢弃工作区中对某个文件的修改，恢复到上次提交的状态：

```bash
# 推荐方式（Git 2.23+）
git restore file.txt

# 传统方式
git checkout -- file.txt

# 恢复所有文件
git restore .
```

注意：这个操作是不可逆的。被丢弃的修改无法找回，因为这些修改从未被提交过。在执行之前，请确保你真的不需要这些修改。

## 6. 🤔 如何使用 `git revert` 进行安全撤销？

`git revert` 通过创建一个新的提交来撤销指定提交的更改，不会修改已有的提交历史，是一种安全的撤销方式：

基本用法：

```bash
# 撤销最近的一次提交
git revert HEAD

# 撤销指定的提交
git revert abc1234

# 撤销多个连续的提交
git revert HEAD~3..HEAD

# 只修改工作区和暂存区，不自动创建提交
git revert --no-commit abc1234
```

对于已推送到远程的提交，应该使用 `git revert` 来撤销，避免对团队协作造成影响。

结论：公共分支严禁随意使用 `reset` 强推。

### 6.1. 核心核心：只新增记录而不修改历史

为什么使用 `git revert` 进行撤销是“安全”的？

`git revert` 的核心逻辑是 “逆向操作” + “新增记录”。

- 不修改历史：它不会删除或修改任何现有的提交（Commit）。
  - Git 的历史记录是一条单向链表，`revert` 只是在链表的末尾追加了一个（或几个）新的提交。
- 生成反向补丁：Git 会分析你指定要撤销的那个提交（比如 commit A），计算出该提交引入了哪些变化（Diff），然后生成一个完全相反的补丁（Reverse Diff），并将其应用为一个新的提交（commit B）。
  - 如果 commit A 增加了某行代码，commit B 就会删除这行代码。
  - 如果 commit A 删除了某个文件，commit B 就会重新创建该文件（恢复内容）。

### 6.2. `git revert` 与 `git reset` 的区别

这是 Git 面试和实际工作中最高频的问题之一。

| 维度 | `git revert` | `git reset` |
| :-- | :-- | :-- |
| 本质操作 | 新增一个提交（正向历史的延续）。 | 移动指针（HEAD/分支指针），可能丢弃提交。 |
| 历史记录 | 保留完整。可以看到“犯错”和“纠错”的全过程。 | 抹除历史。被 reset 掉的提交在本地分支历史中消失（除非用 reflog 找回）。 |
| 适用场景 | 已推送 (Pushed) 到远程的公共分支。 | 未推送的本地私有提交，或整理本地杂乱的历史。 |
| 协作影响 | 无副作用。其他人 pull 后能正常同步撤销操作。 | 破坏性。如果其他人基于被 reset 掉的提交开发了代码，强制推送会导致他们的历史分叉，引发灾难。 |
| 安全性 | ⭐⭐⭐⭐⭐ (极高) | ⭐⭐ (低，尤其是 `--hard`) |
| 典型用途 | 修复线上紧急 Bug、回滚错误的发布版本。 | 本地开发时 commits 写错了、想重组提交历史 (Squash)。 |

- `git reset` 像是用橡皮擦把账本上写错的那一行擦掉，仿佛它从未发生过。（如果别人已经抄了你的账本，他们的账本就和你不一致了）。
- `git revert` 像是在账本新的一行写上：“修正之前的错误，抵消第 X 行的操作”。原来的错误记录还在，但现在的余额是正确的。（所有人都能看到修正记录，账本依然同步）。

#### `git revert`（安全操作）

假设我们有一个主分支 `main`，历史如下：

- C1：初始提交
- C2：添加了功能 A（这是我们要撤销的“错误”提交）
- C3：添加了功能 B（在错误之后又做了新工作）
- HEAD：指向 C3

```text
时间轴：过去 -----------------------------> 现在

初始状态 (执行 git revert C2 之前):

      [C1] ---- [C2] ---- [C3] (HEAD, main)
       |          |          |
     初始版    错误功能 A    新功能 B
```

用户执行命令：

```bash
git revert <C2的Hash>
```

Git 开始工作：

1. 分析 C2：读取 C2 引入的变更（Diff）
2. 生成反向补丁：计算如何抵消这些变更
3. 应用补丁：尝试将反向补丁应用到当前工作区（基于 C3 的状态），如果有冲突，此时会暂停，让用户手动解决
4. 创建新提交：如果没有冲突（或冲突解决后），Git 创建一个新提交 C4

`git revert` 不会删除 C2，也不会移动 C3。它只是在末尾追加了一个新节点 C4。

完成后的历史结构：

```text
执行 git revert C2 之后:

      [C1] ---- [C2] ---- [C3] ---- [C4] (HEAD, main)
       |          |          |          |
     初始版    错误功能 A  新功能 B   撤销功能 A
               (保留)      (保留)     (新增的反向操作)
```

- C2 依然存在：历史记录完整保留了“曾经犯过错”的事实
- C4 的内容：C4 的代码状态 ≈ C1 + C3
  - C2 增加的代码被 C4 删除了
  - C2 删除的代码被 C4 恢复了
  - C3 中添加的“新功能 B”完全不受影响，依然保留
- 线性历史：分支指针 `main` 和 `HEAD` 平滑地向前移动到了 C4，没有发生分叉或重写

#### `git reset`（危险操作）

为了突出 `revert` 的安全性，看看如果用 `reset` 强行回退会发生什么。

```text
如果使用 git reset --hard C1 (并强制推送):

      [C1]
       | \
       |  \-- [C2] ---- [C3] (孤儿节点，即将被垃圾回收)
       |
     (HEAD, main)

后果：
1. C2 和 C3 从分支历史中“消失”了。
2. 如果别人基于 C3 开发了代码，他们的仓库会和你的严重冲突。
3. “新功能 B”的数据在分支视图中丢失了（除非去 reflog 找回）。
```

关于 `git reset` 的危险性说明 => 如果你对一个已经推送到远程的提交使用了 `git reset --hard` 并强制推送 (`git push --force`)：

1. 你的本地历史变短了。
2. 同事的本地历史还保留着旧的提交。
3. 当同事尝试 `git pull` 时，Git 会发现历史分叉，可能导致代码丢失或需要极其复杂的重基（Rebase）操作来修复。

### 6.3. 进阶场景：Revert 多个提交

如果你执行 `git revert C2..C3` (撤销 C2 和 C3)：

Git 会按时间顺序（从旧到新）依次生成反向提交，或者合并为一个。

方式 A：默认（生成多个提交）

```text
[C1] ---- [C2] ---- [C3] ---- [C4] ---- [C5]
  |        |          |        |          |
初始版   错误功能 A  新功能 B  撤销功能 A  撤销功能 B

C4 抵消 C2
C5 抵消 C3
```

方式 B：使用 `--no-commit`（合并为一个提交）

```text
[C1] ---- [C2] ---- [C3] ---- [C4']
  |        |          |        |
初始版   错误功能 A  新功能 B  同时撤销 A 和 B

Git 将 C2 和 C3 的反向变化累积在工作区，最后一次性提交为 C4'。
这样历史更干净，只有一条“回滚”记录。
```

### 6.4. 进阶场景：需要处理 Revert 冲突的情况

在执行 `git revert` 时，如果被撤销的提交修改的文件，在之后的提交中也被修改过，就会产生冲突。

场景：

1. Commit A：添加了函数 `funcX`。
2. Commit B：修改了 `funcX` 的内部逻辑。
3. 现在你想 `revert` Commit A（删除 `funcX`）。
4. Git 困惑了：Commit B 依赖 `funcX`，直接删除 `funcX` 会导致 B 的代码报错。

解决步骤：

```bash
# 1. 执行 revert
git revert <Commit A 的 Hash>
# 输出：CONFLICT (content): Merge conflict in main.js
# 输出：Automatic revert failed; fix conflicts and then commit the result.

# 2. 查看状态
git status
# 显示：both modified: main.js

# 3. 手动解决冲突
# 打开 main.js，找到 <<<<<<, ======, >>>>>> 标记。
# 决定是保留修改后的代码但移除函数定义，还是其他逻辑。
# 编辑保存文件。

# 4. 标记解决
git add main.js

# 5. 完成提交
# 如果之前没用 --no-commit，直接 commit 即可
git commit
# (如果使用 --no-commit，此时手动 commit)
```

### 6.5. 小结

- 黄金法则：只要代码已经 Push 到共享仓库（Origin），永远优先使用 `git revert`。
- 本地清理：只有在代码完全在本地，且没有其他人基于此开发时，才使用 `git reset` 来整理历史（例如将多个琐碎 commit 合并为一个）。
- 原子性：尽量一次 revert 一个逻辑单元。如果需要 revert 一大段历史，建议使用 `--no-commit` 仔细检查后再提交，避免引入难以追踪的 Bug。
- 测试：Revert 完成后，务必运行测试套件。因为“撤销代码”并不总是等于“回到过去的状态”（特别是当中间有其他代码依赖了被撤销的功能时）。
- 沟通：在执行重大 Revert 操作前，最好在团队群里告知一声：“我要 Revert 提交 xxx，因为它导致了线上故障”，以便大家知晓上下文。

这种机制保证了无论何时，只要拉取（pull）最新的代码，所有人的历史线都是统一且向前发展的，永远不会出现“历史凭空消失”的恐怖情况。

```text
+-----------------------+
|   Git Revert 核心逻辑   |
+-----------------------+
| 1. 读取目标提交 (Old)   |
| 2. 计算反向差异 (-Diff) |
| 3. 应用到当前最新代码   |
| 4. 生成新提交 (New)     |
+-----------------------+
          |
          v
    所有历史记录都在
    虽然历史变长了
    但也更加安全了
```
