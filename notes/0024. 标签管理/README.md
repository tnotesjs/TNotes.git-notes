## 2. 🫧 评价

## 3. 🤔 标签是什么？

在 Git 中，标签（Tag）是一个指向特定提交（Commit）的静态引用（指针），是 Git 中用于给某个特定提交（Commit）打上永久性标记的机制，通常用于标记版本发布节点（如 `v1.0.0`、`v2.3.1`）。

与分支不同，标签不会随着新的提交而移动，它始终指向创建时所绑定的那个提交，因此非常适合用来标记一个稳定的、可发布的历史节点。

### 3.1. 使用标签的场景

标签主要用于标记重要的历史节点，最常见的场景包括：

#### 发布软件版本（最常见）

这是标签最主要的用途。当你的项目完成了一个阶段性的开发，准备向用户发布时，你会创建一个标签来标记这个版本。

示例：`v1.0.0`、`v1.2.3-beta`、`release-2024-q1`

作用：

- 用户和开发者可以清楚地知道哪个提交对应哪个版本号
- 如果新版本出现严重 Bug，可以迅速通过标签回退到上一个稳定版本（如从 `v1.1.0` 回退到 `v1.0.0`）
- CI/CD 流水线通常根据标签自动触发构建和部署流程（例如：只有打上 `v*` 标签时才发布到生产环境）

#### 标记关键里程碑

除了正式发版，你也可以用标签标记项目开发过程中的重要时刻。

示例：`prototype-complete`（原型完成）、`security-patch-2024`（安全补丁）、`pre-refactor`（重构前）

作用：方便团队在将来回顾代码演变过程，或者在重构出错时能快速回到重构前的状态。

#### 生成变更日志（Changelog）

许多自动化工具（如 `git-changelog`、`standard-version`）利用标签来计算两个版本之间的差异。

用法：工具会自动查找 `v1.0.0` 和 `v1.1.0` 之间的所有提交记录，并自动生成一份更新说明文档。如果没有标签，这种自动化很难实现。

#### 确保构建的可复现性（Reproducibility）

在生产环境中，必须保证部署的代码是确切知道的版本。

场景：你不能说“部署上周五下午的代码”，因为分支可能已经变了。你必须说“部署 `v2.3.1` 标签对应的代码”。

作用：无论过了多久，只要检出该标签，得到的代码二进制文件是完全一致的，这对于审计和故障排查至关重要。

#### 第三方依赖管理

在某些工作流中，其他项目可能会直接引用你项目的特定标签作为依赖版本，以确保他们使用的库是稳定的，而不是你正在开发中的不稳定分支。

### 3.2. 分支 vs 标签

你可以把标签想象成书本中的“书签”或软件版本历史中的“里程碑”：

- 标签（Tag）是静态的：一旦创建，它就永远固定指向那个特定的提交哈希值（Commit Hash），不会随后续开发而改变。
- 分支（Branch）是动态的：随着你不断提交新代码，分支指针会向前移动（例如 `main` 分支通常指向最新的提交）。

核心作用：给某个特定的历史时刻打上一个人类可读的、有意义的名字（如 `v1.0.0`），方便日后快速找回该版本的代码状态。

| 特性 | 分支（Branch） | 标签（Tag） |
| :-- | :-- | :-- |
| 移动性 | 动态：随新提交自动向前移动 | 静态：创建后永久固定 |
| 用途 | 用于日常开发、功能迭代 | 用于存档、发布、标记里程碑 |
| 命名习惯 | `feature/login`、`bugfix/header` | `v1.0.0`、`v2.1.0-rc1` |
| 是否可修改 | 经常变基（rebase）或合并 | 绝不应修改（删除后重新创建需非常谨慎） |

## 4. 🤔 如何列出标签？

```bash
# 列出所有标签
git tag

# 按通配符模式列出标签
git tag -l "v1.8.*"

# 按版本号排序
git tag --sort=version:refname

# 查看标签详细信息
git show v1.0.0
```

`git tag` 会按字母顺序列出标签。如果标签数量很多，可以使用 `-l` 参数配合通配符来过滤。

## 5. 🤔 Git 中的标签都有哪些类型？

Git 支持两种类型的标签：

- 轻量标签（Lightweight Tag），它只是一个指向特定提交对象（commit object）的简单引用（指针）。
- 附注标签（Annotated Tag），它是一个完整的 Git 对象（tag object），存储在仓库数据库中。

### 5.1. 轻量标签（Lightweight Tag）

轻量标签只是一个指向特定提交的引用，不包含额外信息：

```bash
git tag v1.0.0
```

### 5.2. 附注标签（Annotated Tag）

附注标签是存储在 Git 数据库中的完整对象，包含标签名、邮箱、日期和标签消息，推荐使用：

```bash
# 创建附注标签
git tag -a v1.0.0 -m "发布 1.0.0 版本"
# -a 表示创建附注标签，-m 指定标签消息

# 为历史某个提交打标签
git tag -a v0.9.0 abc1234 -m "补打 0.9.0 版本标签"
```

通常建议使用附注标签，因为它包含了更多的元数据信息，方便日后追溯。

### 5.3. 对比

| 特性 | 轻量标签 (Lightweight) | 附注标签 (Annotated) |
| :-- | :-- | :-- |
| 本质 | 指向提交的简单指针 | 独立的 Git 对象 |
| 元数据 | 无（仅标签名和 Commit ID） | 有（作者、时间、消息、GPG 签名） |
| 创建命令 | `git tag <name>` | `git tag -a <name> -m "msg"` |
| 查看内容 | `git show` 显示提交详情 | `git show` 先显示标签详情，再显示提交详情 |
| 推荐用途 | 临时标记、私有开发标记 | 正式版本发布 (Release) |

## 6. 🤔 如何共享标签到远程仓库？

默认情况下，`git push` 不会将标签推送到远程仓库，需要显式推送：

```bash
# 推送单个标签
git push origin v1.0.0

# 推送所有标签
git push origin --tags

# 只推送附注标签（不推送轻量标签）
git push origin --follow-tags
```

可以设置 `git push` 默认推送附注标签：

```bash
git config --global push.followTags true
```

## 7. 🤔 如何删除标签？

```bash
# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0
# 或
git push origin :refs/tags/v1.0.0
```

删除远程标签需要两步：先删除本地标签，再推送删除操作到远程仓库。

如果想要检出某个标签对应的代码：

```bash
# 这会使 HEAD 进入分离状态
git checkout v1.0.0

# 如果需要在标签基础上开发，创建一个新分支
git switch -c fix-v1.0.0 v1.0.0
```

## 8. 🤔 如何给历史 commit 追加标签？

如果我们忘了给某个提交打标签，又将它发布了，我们可以给它追加标签。

例如，假设我们发布了提交 `85fc7e7...`，但是那时候忘了给它打标签，现在想要给它加上标签 `v1.0.0`：

```bash
git tag v1.0.0 85fc7e7
```
