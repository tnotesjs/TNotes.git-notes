## 2. 🫧 评价

Git 中分支的本质就是一个可变指针，而非仓库全量拷贝后的副本，因此分支操作（创建、切换、...）的开销极低。正因为成本几乎为零，Git 的设计理念就是在团队协作场景中，鼓励开发者为每个新功能、每个 Bug 修复都创建独立的分支，在问题处理完之后再往主分支合并。

关于 Git 分支的操作命令，比如创建、切换、删除、…… 这些都是比较基础的内容，重点在于理解 Git 中的核心概念，比如 HEAD、分支、提交对象、…… 理解了这些，再来看 Git 这些分支管理的命令，就很轻松了。

## 3. 🤔 什么是分支？

在 Git 中，分支本质上是一个指向某个提交对象的可变指针。每次提交时，当前分支的指针会自动向前移动，指向最新的提交。

Git 的默认分支名是 `main`（或 `master`）。

```text
初始状态：分支 'main' 指向提交 C1
    (指针)
      |
      v
    [ C1 ]  <-- (main)


--- 执行第一次提交 (git commit) ---
Git 创建新提交 C2，C2 指向父节点 C1
分支 'main' 的指针自动从 C1 移动到 C2

    [ C2 ]  <-- (main)
      ^
      |
    [ C1 ]


--- 执行第二次提交 (git commit) ---
Git 创建新提交 C3，C3 指向父节点 C2
分支 'main' 的指针再次向前移动，指向最新的 C3

    [ C3 ]  <-- (main)
      ^
      |
    [ C2 ]
      ^
      |
    [ C1 ]


- 提交对象 (C1, C2, C3) 是不可变的，它们通过哈希值链接成链表。
- 分支 (main) 只是一个轻量级的标签/指针。
- "移动"的不是提交，而是分支指针始终指向链表的最新端。
```

分支在 Git 内部的实现非常轻量 —— 它只是一个包含 40 个字符 SHA-1 校验和的文件，指向某个提交对象。因此，创建和销毁分支的成本极低。

```text
Git 分支内部实现原理 (基于 .git/refs/heads/)

1. 分支本质：一个纯文本文件
   路径：.git/refs/heads/main
   内容：e9a3b2c... (40字符的提交哈希值)

   +-----------------------+
   | .git/refs/heads/main  |  <-- 这是一个普通文件
   |-----------------------|
   | e9a3b2c1d4...         |  <-- 仅存储指向最新提交的 SHA-1
   +-----------------------+
            |
            | (读取文件内容)
            v
   +-----------------------+
   |   提交对象 (Commit)   |
   |-----------------------|
   | Tree: a1b2c3...       | (快照)
   | Parent: d4e5f6...     | (父节点)
   | Author: ...           |
   | Message: "fix bug"    |
   +-----------------------+


2. 创建分支 (git branch feature)
   操作：复制 main 文件内容为新文件
   成本：极低 (仅文件系统写操作，无数据复制)

   .git/refs/heads/
   ├── main      [e9a3b2c...]
   └── feature   [e9a3b2c...]  <-- 新文件，指向完全相同的提交


3. 切换分支 (git checkout feature)
   操作：更新 HEAD 文件指向
   成本：低 (更新工作区文件)

   .git/HEAD
   ├── 原内容: ref: refs/heads/main
   └── 新内容: ref: refs/heads/feature  <-- 指针切换


4. 在新分支提交 (git commit on 'feature')
   操作：仅更新 feature 文件中的哈希值
   成本：极低 (修改一行文本)

   提交前:
   .git/refs/heads/feature [e9a3b2c...]

   提交后 (生成新提交 f1g2h3...):
   .git/refs/heads/feature [f1g2h3i...]  <-- 仅修改此文件的一行文字

   此时 main 依然指向 [e9a3b2c...]，互不影响。
```

- 分支 = 一个小文件（可能就是一个 41 字节左右的文件，创建这个文件瞬间就能完成）
- 创建/删除 = 创建/删除一个小文件（瞬间完成）
- 切换 = 修改 HEAD 文件 + 更新工作区（主要耗时在文件 IO，而非分支逻辑）
- 与其他版本控制系统相比，Git 鼓励频繁使用分支，因为分支操作几乎都是即时的，开销几乎可以忽略不计

## 4. 🤔 分支和提交之间的关系是什么？

| 特性 | 提交 (Commit) | 分支 (Branch) |
| :-- | :-- | :-- |
| 本质 | 数据快照（文件树 + 元数据） | 指向提交的指针（文本文件） |
| 可变性 | 不可变（Immutable） | 可变（Mutable），随新提交移动 |
| 存储内容 | 完整的文件内容差异、父节点哈希 | 仅 40 字符的提交哈希值 |
| 生命周期 | 永久存在（除非被 GC 清理且无引用） | 可随时创建、删除、移动 |
| 依赖关系 | 不依赖分支存在 | 强依赖提交存在（不能指向空） |
| 比喻 | 历史书页 / 火车车厢 | 书签 / 站牌 |

### 4.1. 本质

分支是一个指向特定提交的可变指针。

- 提交（Commit）：是 Git 仓库中的实体数据。它是一个不可变的快照，包含文件内容、作者信息、时间戳以及父提交的哈希值。一旦创建，永远不变。
- 分支（Branch）：是 Git 仓库中的轻量级引用。它本质上只是一个文本文件（位于 `.git/refs/heads/`），里面只存了一行字：某个提交的哈希值。

### 4.2. 类比

- 提交就像是火车车厢（实实在在的内容，连成一串）
- 分支就像是贴在某一节车厢上的站牌（比如“北京站”、“上海站”）
- 站牌（分支）本身没有货物，它只是标记了“当前这一列车头停在哪里”

### 4.3. 静止、动态

- 提交是静止的：提交对象一旦生成，它的哈希值和内容的永远不会改变
- 分支是移动的：当你在当前分支上进行新的提交（`git commit`）时
  - Git 创建一个新的提交对象（新节点）
  - 当前分支的指针会自动向前移动，指向这个最新的提交
  - 旧的提交依然保留在历史链中，不再被该分支直接指向（除非有其他分支或标签指向它）

```text
初始状态：
main 分支指向 C1
[ C1 ] <-- (main)

执行一次 commit (生成 C2)：
Git 创建 C2，C2 的父节点是 C1。
main 分支指针从 C1 移动到 C2。
[ C1 ] <- [ C2 ] <-- (main)

再执行一次 commit (生成 C3)：
Git 创建 C3，C3 的父节点是 C2。
main 分支指针继续移动到 C3。
[ C1 ] <- [ C2 ] <- [ C3 ] <-- (main)
```

- 移动的不是提交，而是分支指针
- 提交像铁轨一样铺好就不动了，分支像火车头一样一直往前开

::: info `HEAD detached` 分离头指针

写到这里，感觉正好可以来聊聊分离头指针这玩意儿：

- 正常情况：`HEAD` -> `分支名` -> `提交` => HEAD 指向分支（当前工作的分支），分支指向提交，随着提交增加，分支会不断后移指向最新的提交，同时 HEAD 也会随之后移。
- 分离 HEAD 情况：`HEAD` -> `提交` => HEAD 绕过分支，直接指向提交，此时虽然也能提交，但是由于没有分支指向这些提交，一旦你执行 `git checkout <branch-name>` 切换到其他分支，HEAD 指针就会移走。刚才那些新提交就瞬间变成了“孤儿提交” (Orphan Commit)。Git 的垃圾回收机制（GC）会在一段时间后（默认通常是 30 天或 90 天，取决于配置）自动清理这些无人引用的提交，它们将永久消失。如果你在分离 HEAD 状态下已经提交了，并且想保留这些工作，请立即创建一个新分支 `git checkout -b <新分支名>` 这样就把刚才那些“悬空”的提交正式“接住”并保护起来了。

:::

### 4.4. 对应关系

- 一对一（一个分支只能指向一个提交）
- 多对一（多个分支可以指向同一个提交）

```text
当你创建一个新分支但还没提交时，新分支和旧分支指向同一个提交。

[ C3 ] <-- (main, feature)

此时，main 和 feature 两个分支名都指向 C3 这个提交。它们完全同步。
```

## 5. 🤔 如何创建分支？

使用 `git branch` 命令创建新分支：

```bash
# 创建一个名为 feature 的新分支（不切换）
git branch feature

# 创建新分支并立即切换（推荐）
git switch -c feature
# 或
git checkout -b feature

# 从指定提交创建分支
git branch feature abc1234

# 从远程分支创建本地分支
git switch -c feature origin/feature
```

创建分支时，新分支会指向当前所在的提交。

创建分支并不会自动切换到新分支，除非使用 `git switch -c` 或 `git checkout -b`。

## 6. 🤔 如何切换分支？

```bash
# 推荐方式（Git 2.23+）
git switch feature

# 传统方式
git checkout feature

# 切换到上一个分支
git switch -
```

切换分支时，Git 会将工作区的文件更新为目标分支最新提交的快照。如果工作区有未提交的修改，且与目标分支存在冲突，Git 会阻止切换并提示你先提交或贮藏修改。

`git switch` 是 Git 2.23 引入的新命令，专门用于切换分支，比 `git checkout` 更加语义清晰。

## 7. 🤔 如何查看分支列表？

```bash
# 查看本地分支列表（当前分支前有 * 标记）
git branch

# 查看本地分支及最后一次提交信息
git branch -v

# 查看所有分支（包括远程分支）
git branch -a

# 只查看远程分支
git branch -r

# 查看已合并到当前分支的分支
git branch --merged

# 查看未合并到当前分支的分支
git branch --no-merged
```

`--merged` 和 `--no-merged` 在清理过时分支时特别有用。已合并的分支通常可以安全删除。

## 8. 🤔 `HEAD` 指针是什么？

### 8.1. `HEAD` 指针

`HEAD` 是一个特殊的指针，指向当前所在的本地分支。可以把 `HEAD` 理解为“当前分支的别名”。

当你切换分支时，`HEAD` 会指向新的分支：

```bash
git switch feature
# HEAD 现在指向 feature 分支
```

当你在当前分支提交时，`HEAD` 指向的分支会自动向前移动。

### 8.2. 查看 `HEAD` 指向

可以通过以下命令查看 `HEAD` 指向的分支：

```bash
# 查看 HEAD 指向的分支（推荐方式）
git symbolic-ref HEAD
# 输出: refs/heads/main

# 如果你只想要分支名（如 main 而不是 refs/heads/main），可以配合 --short 参数
git symbolic-ref --short HEAD
# 输出: main

# 当你处于 Detached HEAD 状态时，git symbolic-ref HEAD 会返回非零退出码并报错
git symbolic-ref HEAD
# 输出: fatal: ref HEAD is not a symbolic ref

# 查看 HEAD 指向的分支（其它做法）
cat .git/HEAD
# 输出: ref: refs/heads/main
```

可以通过以下命令直接查看 HEAD 指向的具体提交哈希值：

```bash
git rev-parse HEAD
# 输出：df1d06f8c46b48a147688fa04a967acb87fcfc33
```

### 8.3. 分离头指针 `HEAD detached`

分离 HEAD 状态 `HEAD detached` 是指 `HEAD` 指针不再指向某个分支名称（如 `main` 或 `feature`），而是直接指向了一个具体的提交对象（Commit Hash）。

在这种状态下，你依然可以查看代码、甚至进行修改和提交，但没有分支指针跟随你的新提交移动。

在 `HEAD detached` 状态下提交的代码，如果不立即创建新分支保存，一旦切换分支，这些提交就会像断了线的风筝一样飞走（丢失）。

| 特性 | 正常分支模式 | 分离 HEAD 模式 |
| :-- | :-- | :-- |
| HEAD 指向 | 指向分支名 (ref: refs/heads/main) | 直接指向 Commit Hash |
| 提交行为 | 分支指针随提交自动前移 | 无分支指针移动，提交悬空 |
| 切换后果 | 安全，提交被保留 | 若未创建分支，新提交会丢失 |
| 适用场景 | 日常开发、功能迭代 | 临时查看历史、调试、CI 构建 |
| 建议 | 默认工作模式 | 仅用于临时操作，若要保留成果必须新建分支 |

### 8.4. 导致发生分离头指针的常见场景

这种情况通常发生在检出特定提交、标签，或合并冲突处理中，比如：

1. 检出历史提交：你想查看或测试过去的某个版本。

```bash
git checkout abc1234  # 直接检出一个哈希值
```

2. 检出标签（Tag）：标签通常是固定的，检出标签也会进入分离状态。

```bash
git checkout v1.0.0
```

3. 合并冲突处理中：在某些复杂的合并或变基（rebase）操作中，Git 可能会临时进入此状态。

### 8.5. 分离头指针的核心风险：提交可能会“丢失”

这是理解 Detached HEAD 最关键的一点。

- 正常模式：`HEAD` -> `分支名` -> `提交C`。当你 commit 时，`提交C` 更新，`分支名` 随之移动，`HEAD` 也跟着移动。你的成果被分支名“记住”了。
- 分离模式：`HEAD` -> `提交C`。当你 commit 生成 `提交D` 时，`HEAD` 会移动到 `提交D`，但是没有任何分支名指向 `提交D`。

后果：如果你此时切换回其他分支（例如 `git switch main`），由于没有分支指向刚才在分离状态下创建的 `提交D`，它很快就会变成“孤儿节点”，最终被 Git 的垃圾回收机制（GC）清理掉，导致代码丢失。

```text
场景演示：

1. 当前在 main 分支 (指向 C2)
   [C1] <- [C2] <-- (main, HEAD)

2. 执行 git checkout C1 (进入分离 HEAD 状态)
   [C1] <-- (HEAD)
     ^
     |
   [C2] <-- (main)
   (注意：HEAD 直接指着 C1，而不是 main)

3. 在分离状态下修改代码并提交，生成 C3
   [C1] <- [C3] <-- (HEAD)
     ^
     |
   [C2] <-- (main)
   (注意：C3 没有分支名指向它！只有 HEAD 指着它)

4. 危险操作：切换回 main 分支
   git switch main

   结果：
   [C1] <- [C3] (孤儿节点，无人引用，即将被 GC 回收)
     ^
     |
   [C2] <-- (main, HEAD)

   ❌ C3 中的代码修改实际上已经“丢失”了（除非你去 reflog 找回）
```

### 8.6. 安全地处理分离 HEAD 状态

如果你发现自己处于分离 HEAD 状态，并且做了一些重要的修改或提交，务必先创建一个新分支来“接住”这些提交，然后再切换走。

操作步骤：

```bash
# 1. 发现当前是分离 HEAD 状态 (git status 会提示)
# HEAD detached at abc1234

# 2. 基于当前状态创建一个新分支，保存成果【关键步骤】
git switch -c my-temp-fix
# 或者
# git checkout -b my-temp-fix

# 此时：
# HEAD -> my-temp-fix -> 当前提交
# 你的提交现在被分支名保护起来了，不会再丢失！

# 3. 现在可以安全地切换回主分支或其他分支
git switch main
```

### 8.7. 刻意使用分离 HEAD 的场景

虽然分离 HEAD 听起来很危险，但它也确实是非常有用的临时探索工具，我们可以利用它会丢失提交记录的特点来实现一些比较特殊的需求：

- 验证旧版本：快速 checkout 一个旧版本，运行测试，确认 Bug 是否存在。验证完后直接切回主分支即可（不需要保存修改）。
- 构建特定版本：CI/CD 流程中经常需要 checkout 特定的 Tag 或 Commit 进行打包，而不需要创建新分支。
- 交互式变基（Rebase）：在执行 `git rebase -i` 时，Git 会临时进入分离 HEAD 状态来逐个应用提交。

根据这篇笔记的内容，以下是 Git 分支机制 中面试最高频的考点，按重要性排序：

### 9.1. 🌟 核心概念：分支的本质 (必考)

- 考点：Git 中的分支到底是什么？
- 答案要点：
  - 分支不是一个文件夹副本，而是一个指向特定提交（Commit）的可变指针
  - 它在内部只是一个包含 40 字符 SHA-1 哈希值的小文件（位于 `.git/refs/heads/`）
  - 优势：因为只是移动指针，所以创建、切换、删除分支的开销极低（几乎瞬时完成），这也是 Git 鼓励“频繁使用分支”工作流的原因

### 9.2. 🌟 HEAD 指针与分离头指针 (Detached HEAD) (高频难点)

- 考点 A：`HEAD` 是什么？
  - 答案：`HEAD` 是一个特殊指针，指向当前所在的本地分支。正常流程是 `HEAD` -> `分支名` -> `提交`。
- 考点 B：什么是“分离头指针”（Detached HEAD）？怎么发生的？
  - 答案：当 `HEAD` 不再指向分支名，而是直接指向某个具体的提交哈希时，就是分离头指针状态。
  - 常见场景：检出历史提交 (`git checkout <hash>`) 或检出标签 (`git checkout <tag>`)。
- 考点 C：在分离头指针状态下提交代码会发生什么？（陷阱题）
  - 答案：可以提交，但新提交没有分支名指向它。一旦切换回其他分支，这些提交就会变成“孤儿”，最终被 Git 垃圾回收（GC）清理，导致代码丢失。
  - 补救措施：如果要在该状态下保留工作，必须立即基于当前状态创建一个新分支 (`git switch -c <新分支名>`) 来“接住”这些提交。

### 9.3. 🔄 分支与提交的关系 (基础理解)

- 考点：提交时，是谁在移动？
  - 答案：提交对象是不可变的（静止的），移动的是分支指针。
  - 每次 `git commit`，Git 创建一个新的提交对象，然后将当前分支的指针向前移动到新的提交上。
- 考点：多个分支可以指向同一个提交吗？
  - 答案：可以（多对一）。例如刚创建新分支但未提交时，新旧分支指向同一个提交。

### 9.4. 🛠️ 常用命令与参数 (实操考察)

- 创建并切换分支：
  - 旧命令：`git checkout -b <branch>`
  - 新命令（推荐）：`git switch -c <branch>` (Git 2.23+)
- 查看分支：
  - `git branch -v`：查看本地分支及最后一次提交信息。
  - `git branch --merged` / `--no-merged`：清理分支时的神器。用于找出哪些分支已合并（可安全删除）或未合并（需小心处理）。
- 查看 HEAD 指向：
  - `git symbolic-ref HEAD`：查看当前分支名（分离状态下会报错）。
  - `cat .git/HEAD`：查看底层文件内容。

### 9.5. 💡 面试官可能的追问剧本

1.  “为什么 Git 切换分支这么快？”
    - 答：因为只是修改 `.git/HEAD` 文件和更新工作区文件，不需要复制整个项目数据。
2.  “我不小心 `checkout` 了一个 commit hash，改完代码想切回 main，怎么办才不丢代码？”
    - 答：先 `git switch -c temp-branch` 保存现场，再 `git switch main`，最后可以把 `temp-branch` 合并或变基到 main。
3.  “`git branch --merged` 有什么用？”
    - 答：用于识别那些已经合并到当前分支的功能分支，这些分支通常可以安全删除，保持仓库整洁。

掌握以上几点，基本就能覆盖 90% 关于 Git 分支机制的面试题。
