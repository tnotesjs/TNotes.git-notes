## 3. 🤔 变基的基础用法是什么？

变基（Rebase）是将一个分支上的提交"移植"到另一个分支上的操作。它会将当前分支的提交重新应用到目标分支的最新提交之后，从而得到一条线性的提交历史。

```bash
# 将当前分支变基到 main 分支
git switch feature
git rebase main
```

变基前：

```
          C3 -> C4 (feature)
         /
C1 -> C2 -> C5 (main)
```

变基后：

```
C1 -> C2 -> C5 (main) -> C3' -> C4' (feature)
```

注意变基后的 `C3'` 和 `C4'` 是全新的提交（SHA-1 值不同），虽然内容与原来相同。

变基的另一个常用方式是指定 `--onto` 参数：

```bash
# 将 feature 分支中从 topic 分叉后的提交变基到 main
git rebase --onto main topic feature
```

## 4. 🤔 变基和合并有什么区别？

变基和合并都是整合分支的方式，但结果不同：

| 特性     | 变基（Rebase）   | 合并（Merge）  |
| -------- | ---------------- | -------------- |
| 提交历史 | 线性，整洁       | 保留分支结构   |
| 原理     | 重写提交         | 创建合并提交   |
| 安全性   | 会改变提交 SHA-1 | 不改变已有提交 |
| 适用场景 | 本地分支整理     | 公共分支整合   |

合并：

```
          C3 ---\
         /       \
C1 -> C2 -> C5 -> C6 (merge commit)
```

变基：

```
C1 -> C2 -> C5 -> C3' (线性历史)
```

选择建议：

- 本地未推送的分支 → 使用变基保持历史整洁
- 已推送到远程的公共分支 → 使用合并保证安全
- 拉取远程更新时 → 推荐使用 `git pull --rebase`

## 5. 🤔 变基有哪些风险？

变基最重要的原则：不要对已推送到公共仓库的提交执行变基。

变基会重写提交历史（改变 SHA-1 值）。如果你变基了已经推送的提交，其他基于这些提交工作的人将会遇到问题——他们的提交历史会与远程不一致，必须进行复杂的合并才能修复。

危险场景示例：

```bash
# 在 main 分支上变基 —— 不要这样做！
git switch main
git rebase feature
git push --force  # 这会覆盖远程历史，影响所有人
```

安全的变基原则：

- 只对本地未推送的提交进行变基
- 如果必须推送变基后的代码，使用 `--force-with-lease`
- 在个人功能分支上变基是安全的（只要没有其他人基于你的分支工作）

## 6. 🤔 如何使用变基解决冲突？

变基过程中如果遇到冲突，Git 会暂停并让你手动解决：

```bash
git rebase main
# 遇到冲突时，Git 会停下来

# 1. 查看冲突文件
git status

# 2. 手动解决冲突（编辑文件，删除冲突标记）

# 3. 将解决后的文件标记为已解决
git add resolved_file.txt

# 4. 继续变基过程
git rebase --continue
```

变基过程中的其他操作：

```bash
# 跳过当前有冲突的提交
git rebase --skip

# 放弃变基，回到变基前的状态
git rebase --abort
```

与合并冲突不同，变基可能需要逐个提交地解决冲突（因为变基是逐个重放提交的）。如果冲突较多，可以考虑先合并再处理。
