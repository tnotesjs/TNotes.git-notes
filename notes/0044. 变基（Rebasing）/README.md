## 2. 🫧 评价

变基是 Git 中通过“重写历史”来整理、移动或合并提交，从而构建线性开发轨迹的手段。

就好比你拿着一把剪刀修剪 Git 的历史 commit，这些修剪下来的 commit，你可以做以下操作：

| 操作 | 描述 |
| --- | --- |
| 丢弃 `Drop` | 直接丢弃（删除） |
| 重写 `Reaord` | 重新装饰再接上（重写提交信息） |
| 合并 `Squash/Fixup` | 将该 commit 和前一个 commit 接到一起形成一个新的 commit（合并） |
| 移植/线性化 `Rebase onto` | 将某个分支上的一段 commit 接到另一个分支上，形成线性的历史 |

Git 变基相关的内容，也是面试中的常客。

## 3. 🤔 变基是什么？

Git 中的变基（Rebase）是一个强大但需要谨慎使用的功能，是将一个分支上的提交“移植”到另一个分支上的操作。

变基会将当前分支的提交重新应用到目标分支的最新提交之后，从而得到一条线性的提交历史。

变基的核心目的是重写提交历史，使项目历史看起来是线性的，仿佛你的工作是基于最新代码直接开发的，而不是在一个旧的时间点上分叉出来的。

## 4. 🤔 `git rebase` 的基本工作流程是？

下面我们来认识一下变基的基本原理。

当你执行 `git rebase` 时，Git 实际上在执行以下步骤：

1. 找到共同祖先：找到当前分支和目标分支的共同提交点。
2. 保存补丁：将当前分支上独有的每一个提交（Commit）保存为临时的补丁文件（patch）。
3. 重置指针：将当前分支的指针移动到目标分支的最新提交上。
4. 应用补丁：按顺序将这些补丁一个一个地应用到新的基底上。
5. 生成新提交：每应用一个补丁，就会生成一个新的提交对象（因此 SHA-1 哈希值会改变）。

结果：提交历史变成了一条直线，没有分叉线。

## 5. 🤔 变基都有哪些常见的类型？

从使用场景来划分的话，可以将变基分为三种：

- 标准变基（保持线性历史）
- 交互式变基（整理提交历史）
- 高级用法 `--onto`（手术刀式变基）

## 6. 🤔 如何实现标准变基，保持线性历史？

这是最常见的用法，用于在开发功能分支时，同步主分支的最新更改。

比如，你的 Git 项目当前的分支状态如下：

```
          C3 -> C4 (feature)
         /
C1 -> C2 -> C5 (main)
```

此时可以通过以下命令实现标准变基，保持线性历史：

```bash
# 1. 切换到功能分支
git switch feature

# 2. 将 feature 变基到 main
git rebase main
```

下面我们来看看命令 `git rebase main` 执行之后都发生了什么：

- 生成补丁文件 patch：Git 会把 `feature` 分支上独有的提交（如 `C3`、`C4`）暂时“拿走”
- 改变指针指向：把 `feature` 的指针指向 `main` 的最新提交（`C5`）
- 应用补丁 path：把“拿走”的提交重新应用在 `C5` 之后，生成 `C3'`、`C4'`

变基后：

```
C1 -> C2 -> C5 (main) -> C3' -> C4' (feature)
```

注意：变基后的 `C3'` 和 `C4'` 是全新的提交（SHA-1 值不同），虽然内容与原来相同。

作用：

- 历史记录清晰整洁，像是一条单线发展的故事
- 便于使用 `git bisect` 等工具排查问题

## 7. 🤔 如何实现交互式变基，整理提交历史？

在将代码合并到主分支之前，开发者常用来“打扫”自己的提交记录（如合并琐碎提交、修改提交信息、删除错误提交）。

```bash
# 启动交互模式，整理最近 3 个提交
git rebase -i HEAD~3
```

执行后会打开编辑器，你可以对提交进行以下操作：

| 操作 | 作用 |
| --- | --- |
| `pick` | 保留该提交 |
| `reword` | 保留提交，但修改提交信息（commit message） |
| `squash` | 将该提交合并到前一个提交中（常用于将 "fix typo", "add logic" 合并为一个完整的 "add feature" 提交） |
| `drop` | 删除该提交 |

## 8. 🤔 如何使用 `--onto` 参数，实现手术刀式变基？

`--onto` 允许你非常精确地控制哪些提交需要被移动，以及移动到哪里。

命令结构：`git rebase --onto <新基底> <上游参考点> <当前分支>`

经典案例：从错误的分支分叉了

假设你有三个分支：

```
       C -> D (server 你在该分支上开发了新功能)
      /
A -> B (main)
     \
      E (client)
```

### 8.1. 实践：将 B 到 D 这段分支“嫁接”到 E 后边儿

如果你想把 `server` 分支中从 `B` 之后的提交（即 `C` 和 `D`），移植到 `client` 分支的顶端（而不是整个 server 分支合并过来），可以执行以下命令：

```bash
git rebase --onto client main server
```

- `--onto client`：新的基底（就是被接的分支）是 client 分支（指向 E）
- `main`：游参考点（指向 B），Git 会排除 B 及其之前的所有提交，只保留 B 之后的部分（即 C、D）
  - 这一步骤相当于明确你要剪的是哪一部分
  - => `(main, server]`，从 main 开始剪（不包括 main，也就是 B）到 server 停止（包括 server，也就是 D）
  - => 即 C、D
- `server`：要操作的分支（就是被剪的分支）

经过上述变基之后的结果：`server` 分支的历史变成了 `A -> B -> E -> C' -> D'`。

```
       C -> D (C 和 D 变成了悬空提交，它们会在大约 30 天后被 Git 的垃圾回收机制 git gc 永久清除)
      /
A -> B (main)
     \
      E -> C' -> D'(server, client)
```

注意：`B` 之前的提交不会被重复应用，只有 `C` 和 `D` 被“剪切”并“粘贴”到了 `client` (E) 后面，生成了新的提交 `C'` 和 `D'`。此时 `server` 分支指针指向 `D'`。

### 8.2. 实践：将 B 到 D 这段分支直接剪掉

初始情况：

```text
      C --- D  (server 分支，指向 D)
     /
A --- B        (main 分支，指向 B)
     \
      E        (client 分支，指向 E)
```

如果需要将 C、D 直接剪掉（丢弃），可以执行以下命令：

```bash
git rebase --onto client server server
```

这个命令本质上改变的是 `server` 分支的指向，`client` 分支本身不受影响，依然指向 E。

流程分析：

1. 操作对象：`server` 分支
2. 上游参考点：`server` 提交 D
3. 计算差集：找出 `server` 分支上有，但 `server` D 及其祖先中没有的提交
4. 结果：空集（没有任何提交在 D 之后）
5. 动作：将 `server` 分支的指针直接重置到 `--onto` 指定的目标，即 `client` E

最终历史：`server` 分支现在指向 `E`。历史线是 `A -> B -> E`。C 和 D 被丢弃了。

以下是执行之后的状态：

```text
      C --- D  (悬空)
     /
A --- B        (main 分支，指向 B)
     \
      E        (server, client 分支都指向 E)
```

## 9. 🤔 变基和合并有什么区别？

变基和合并都是整合分支的方式，但结果不同：

```
          C3 -> C4 (feature)
         /
C1 -> C2 -> C5 (main)

合并：
          C3 -> C4 --\
         /            \
C1 -> C2 -> C5 -----> C6 (有 merge commit 分叉)

变基：
C1 -> C2 -> C5 -> C3' -> C4' (线性历史，无 merge commit 分叉)
```

| 特性 | 变基（Rebase） | 合并（Merge） |
| --- | --- | --- |
| 原理 | 重写提交 | 创建合并提交 |
| 安全性 | 会改变提交 SHA-1，有风险，属于破坏性操作 | 不改变已有提交的哈希值，相对安全，属于非破坏性操作 |
| 真实性 | 重写了历史，假装你是基于最新代码开发的（虽然实际上不是） | 忠实记录了代码实际发生的整合过程（何时分叉，何时合并） |
| 提交历史 | 线性，整洁 => 历史形状直接强制线性，无 Merge Commit 分叉，就是一条直线 | 保留分支结构 => 合并历史位置会有 Merge Commit 分叉结构，形成非线性的网状结构 |
| 适用场景 | 本地分支整理 => 本地私有分支、整理个人提交历史、同步上游代码 | 公共分支整合 => 公共分支、发布分支、多人协作的主干 |

选择建议：

- 本地未推送的分支 -> 使用变基保持历史整洁
- 已推送到远程的公共分支 -> 使用合并保证安全
- 拉取远程更新时 -> 推荐使用 `git pull --rebase`

## 10. 🤔 实战建议：为什么拉取远程更新时建议使用 `git pull --rebase`？

在多人协作的开发环境中，当你准备拉取远程仓库的最新代码时（例如执行 `git pull`），默认行为是执行 `Fetch + Merge`。虽然这能保证代码同步，但会在本地历史中产生大量无意义的“合并提交”（Merge Commit）。

推荐使用 `git pull --rebase`（即 `Fetch + Rebase`），主要是为了保持提交历史的线性整洁，避免历史记录变成杂乱的“铁轨”或“气泡”状。

### 10.1. 默认行为：`git pull` = Fetch + Merge

当你直接运行 `git pull` 时，如果远程有更新，且你本地也有新提交，Git 会自动创建一个合并节点来连接两条线。

场景假设：

- 远程 `main` 更新了：`A -> B -> R1`
- 你本地基于 `B` 开发了：`A -> B -> C1`

执行 `git pull` 后：

```text
      C1 (你的本地提交)
     /  \
... B    M (自动生成的 "Merge branch..." 提交)
     \  /
      R1 (远程的新提交)
```

这种行为会导致的问题：

- 历史杂乱：每次同步都会产生一个 `M` 节点。如果一天同步多次，历史树会充满这种无实际代码变更的“气泡”。
- 难以回溯：在使用 `git log --oneline` 或 `git bisect` 排查问题时，这些多余的合并节点会干扰视线，让真正的功能迭代路径变得不清晰。

### 10.2. 推荐行为：`git pull --rebase` = Fetch + Rebase

使用 `--rebase` 参数后，Git 会先将你的本地提交“暂时拿走”，把当前分支指针移动到远程最新位置，然后再把你的提交重新应用在后面。

执行 `git pull --rebase` 后：

```text
... B -> R1 -> C1'
```

你的本地提交会直接被“嫁接”到 R1 的后面

注：

- 这个环节如果存在冲突，那还是需要手动处理的
- 上述的 `C1'` 是新的提交哈希值（而非你本地的 C1），因为它的父节点从 `B` 变成了 `R1`

使用这种推荐方案的好处：

- 线性历史：历史记录看起来就像是你是在拉取完最新代码之后才开始开发的一样，是一条完美的直线。
- 逻辑清晰：真实反映了功能的迭代顺序，没有无关的分叉结构。
- 便于审查：Code Review 时，审查者可以看到清晰的功能提交序列，而不需要跳过一堆合并提交。

### 10.3. 对比总结

| 特性     | `git pull` (Merge)               | `git pull --rebase`            |
| :------- | :------------------------------- | :----------------------------- |
| 操作逻辑 | 创建一个新的合并提交节点         | 重写本地提交的基底（变基）     |
| 历史形状 | 非线性，多分叉（像铁轨/气泡）    | 线性，单一直线                 |
| 提交哈希 | 本地提交哈希不变                 | 本地提交哈希会改变             |
| 适用场景 | 公共分支合并、需要保留完整拓扑时 | 个人开发分支同步远程、日常开发 |

### 10.4. 如何配置永久生效？

如果你希望每次执行 `git pull` 时都默认使用 rebase 模式，无需每次都输入 `--rebase`，可以进行全局配置：

```bash
# 全局配置（推荐）
git config --global pull.rebase true

# 或者仅针对当前仓库配置
git config pull.rebase true
```

配置完成后，直接运行 `git pull` 即可享受线性的提交历史。

## 11. 🤔 变基有哪些风险？

变基最重要的原则：不要对已推送到公共仓库的提交执行变基。

变基会重写提交历史（改变 SHA-1 值）。如果你变基了已经推送的提交，其他基于这些提交工作的人将会遇到问题 => 他们的提交历史会与远程不一致，必须进行复杂的合并才能修复。

危险场景示例：

```bash
# 在 main 分支上变基 => 不要这样做！！！
git switch main
git rebase feature
git push --force  # 这会覆盖远程历史，影响所有人
```

安全的变基原则：

- 只对本地未推送的提交进行变基
- 如果必须推送变基后的代码，使用 `--force-with-lease`
- 在个人功能分支上变基是安全的（只要没有其他人基于你的分支工作）

## 12. 🤔 如何使用变基解决冲突？

变基过程中如果遇到冲突，Git 会暂停并让你手动解决：

```bash
git rebase main
# 遇到冲突时，Git 会停下来

# 1. 查看冲突文件
git status

# 2. 手动解决冲突（编辑文件，删除冲突标记）

# 3. 将解决后的文件标记为已解决
git add resolved_file.txt

# 4. 继续变基过程
git rebase --continue
```

变基过程中的其他操作：

```bash
# 跳过当前有冲突的提交
git rebase --skip

# 放弃变基，回到变基前的状态
git rebase --abort
```

与合并冲突不同，变基可能需要逐个提交地解决冲突（因为变基是逐个重放提交的）。如果冲突较多，可以考虑先合并再处理。

### 13.1. 核心概念与原理

Q: 请简述 Git 中 `rebase` 和 `merge` 的区别？

- 核心差异：
  - Merge (合并)：创建一个全新的“合并提交”（Merge Commit），保留完整的分支拓扑结构（非线性历史，有分叉）。它是非破坏性的，不改变已有提交的 Hash。
  - Rebase (变基)：将当前分支的提交“剪切”并“粘贴”到目标分支的最新提交之后，形成一条线性历史。它是破坏性的，会重写提交历史，生成新的提交对象（Hash 值改变）。
- 适用场景：
  - `Merge`：用于公共分支（如 `main`, `develop`）的集成，保留真实的历史发生过程。
  - `Rebase`：用于本地私有分支整理提交记录，或在同步远程代码时保持历史整洁。

---

Q: `git rebase` 的底层工作原理是什么？

三步走：

1. 找祖先：找到当前分支和目标分支的共同祖先。
2. 存补丁：将当前分支独有的提交保存为临时补丁（patch）。
3. 重放：将目标分支指针移到最新位置，然后按顺序逐个应用补丁，生成新的提交（因此 Hash 会变）。

---

### 13.2. 场景应用与最佳实践

Q: 为什么推荐在拉取远程代码时使用 `git pull --rebase`？

- 原因：默认的 `git pull` 是 `fetch + merge`，会在每次同步时产生无意义的“合并提交”，导致历史树杂乱（像铁轨或气泡）。
- 优势：`git pull --rebase` (`fetch + rebase`) 会将本地提交“嫁接”到远程最新提交之后，保持线性历史，便于 `git log` 查看和 `git bisect` 排查问题。
- 配置：可通过 `git config --global pull.rebase true` 永久生效。

---

Q: 如何整理混乱的本地提交历史（如合并琐碎 commit、修改提交信息）？

- 命令：使用交互式变基 `git rebase -i <commit-hash>` (例如 `HEAD~3`)。
- 常用操作指令：
  - `pick`：保留。
  - `reword`：保留提交但修改 message。
  - `squash` / `fixup`：将当前提交合并到前一个提交中（消除琐碎 commit）。
  - `drop`：直接删除该提交。

Q: 如何实现“手术刀式”的变基？（高级用法 `--onto`）

- 场景：从错误的分支分叉了，只想移动某一段特定的提交到另一个分支。
- 命令结构：`git rebase --onto <新基底> <上游参考点> <当前分支>`
- 逻辑解析：
  - Git 会选取 `(上游参考点, 当前分支]` 区间内的提交（即不包含参考点，包含当前分支顶端）。
  - 将这些提交移植到 `<新基底>` 之后。
- 经典案例：
  - 把 `server` 分支上基于 `main` 开发的 `C, D` 提交，移植到 `client` 分支后： `git rebase --onto client main server`

### 13.3. 风险控制（高频陷阱题）

Q: 变基（Rebase）最大的风险是什么？什么情况下绝对禁止使用？

- 黄金法则：永远不要对已经推送到公共仓库（Public Repository）且其他人可能基于此工作的分支进行变基。
- 原因：
  - Rebase 会改变提交的 SHA-1 Hash 值。
  - 如果你强制推送（`git push --force`）了变基后的历史，其他同事的本地历史将与远程不一致，导致他们无法正常合并，甚至丢失工作成果。
- 安全原则：
  - 仅对本地未推送的提交进行变基。
  - 如果必须覆盖远程历史，使用 `git push --force-with-lease`（比 `--force` 更安全，会检查远程是否有新提交）。

### 13.4. 实操与冲突解决

Q: 在 `git rebase` 过程中遇到冲突怎么办？

- 流程：
  1. Git 会暂停在冲突的提交处。
  2. 手动编辑文件解决冲突。
  3. 执行 `git add <file>` 标记解决。
  4. 执行 `git rebase --continue` 继续应用下一个补丁。
- 其他命令：
  - `git rebase --skip`：跳过当前有冲突的提交（慎用，会丢失该提交内容）。
  - `git rebase --abort`：放弃整个变基过程，恢复到变基前的状态（这是 Rebase 相比 Merge 的一个优势，容易“悔棋”）。
- 难点：不同于 Merge 只需解决一次冲突，Rebase 可能需要逐个提交地解决冲突。

### 13.5. 💡 面试回答技巧总结

1. 先讲区别：开口先说 "Merge 保留历史拓扑，Rebase 创造线性历史"。
2. 再讲场景：强调 "本地开发用 Rebase 整理，公共合并用 Merge 保真"。
3. 必谈风险：主动提及 "严禁对公共分支 Rebase"，这能体现你的团队协作意识和严谨性。
4. 举例说明：如果能顺口说出 `git pull --rebase` 的好处或者 `rebase -i` 的用法，会是很大的加分项。
