## 3. 🤔 冲突是怎么产生的？

当 Git 无法自动合并两个分支的修改时，就会产生合并冲突。最常见的原因是两个分支修改了同一个文件的同一部分。

Git 能够自动合并的情况：

- 两个分支修改了不同的文件
- 两个分支修改了同一个文件的不同部分

Git 无法自动合并的情况：

- 两个分支修改了同一个文件的同一行
- 一个分支修改了文件，另一个分支删除了该文件
- 两个分支都添加了同名但内容不同的新文件

## 4. 🤔 如何查看冲突文件？

合并发生冲突后，Git 会停止合并过程并标记冲突文件。使用 `git status` 查看冲突状态：

```bash
git status
```

输出会显示 `Unmerged paths` 部分，列出所有冲突文件，标记为 `both modified`。

冲突文件中，Git 会用特殊标记标识冲突区域：

```
<<<<<<< HEAD
这是当前分支（HEAD）的内容
=======
这是要合并进来的分支的内容
>>>>>>> feature
```

- `<<<<<<< HEAD` 到 `=======` 之间是当前分支的内容
- `=======` 到 `>>>>>>> feature` 之间是被合并分支的内容

## 5. 🤔 如何手动解决冲突？

解决冲突的步骤：

1. 打开冲突文件，找到冲突标记
2. 决定保留哪部分内容（或者合并两者）
3. 删除冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`）
4. 保存文件

例如，将冲突区域修改为最终需要的内容：

```javascript
// 解决冲突后的最终代码
function greet(name) {
  return `Hello, ${name}! Welcome back.`
}
```

解决完所有冲突文件后，将文件标记为已解决：

```bash
git add resolved_file.txt
```

## 6. 🤔 如何使用合并工具解决冲突？

Git 支持调用外部合并工具来可视化地解决冲突：

```bash
# 使用配置的默认合并工具
git mergetool

# 使用指定的合并工具
git mergetool --tool=vimdiff
```

配置默认的合并工具：

```bash
# 设置 VS Code 为合并工具
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# 设置不保留 .orig 备份文件
git config --global mergetool.keepBackup false
```

常见的合并工具包括：VS Code、IntelliJ IDEA、Beyond Compare、KDiff3、Meld 等。现代编辑器通常内置了冲突解决界面，可以直接点击选择保留哪部分的内容。

## 7. 🤔 冲突解决后如何提交？

所有冲突解决完毕后，需要完成合并提交：

```bash
# 将所有解决后的文件标记为已解决
git add .

# 完成合并提交
git commit
```

Git 会自动生成一个包含冲突信息的提交消息，你可以直接使用或编辑后保存。

如果在解决冲突的过程中想要放弃合并，回到合并前的状态：

```bash
git merge --abort
```

这会撤销合并操作，将工作区恢复到合并前的状态。
